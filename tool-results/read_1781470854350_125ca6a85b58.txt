    26→## Contexte & Philosophie
    27→
    28→### Contexte métier
    29→
    30→Application PHP de gestion de workflows de validation de formulaires pour la **DREETS** (Direction Régionale de l'Économie, de l'Emploi, du Travail et des Solidarités) — administration publique française, région Bourgogne-Franche-Comté.
    31→
    32→Hébergée sur **Windows Server 2025 bare metal**, **IIS**, **PHP 8 FastCGI**, **SQLite**.
    33→Auth Windows (Kerberos) fournie par IIS — `$_SERVER['AUTH_USER']` = `DOMAINE\login`.
    34→
    35→### Répertoire de base : `C:\inetpub\wwwroot\workflow\`
    36→
    37→Tous les fichiers PHP sont dans ce répertoire racine. Il n'y a aucun sous-dossier de code.
    38→
    39→### Philosophie — à respecter absolument
    40→
    41→| Principe | Description |
    42→|---|---|
    43→| **Zéro framework PHP** | Pas de Laravel, Symfony, Slim. PHP procédural pur. |
    44→| **Zéro JavaScript** | Aucun JavaScript depuis la v3.0.0. Toutes les interactions sont côté serveur (PHP POST/GET, CSS). Les onglets utilisent des paramètres GET, les confirmations passent par une page dédiée, les toggles sont des liens directs. |
    45→| **Zéro CDN** | Aucune ressource externe. Tout est local. |
    46→| **Zéro dépendance inutile** | PHPMailer est la seule dépendance. Ne pas en ajouter. |
    47→| **Future-proof** | Le code doit tourner sans modification dans 10 ans. Éviter toute API ou syntaxe susceptible d'être dépréciée. |
    48→| **KISS** | Chaque fichier fait une chose. Pas d'abstraction inutile. |
    49→| **CSS partagé** | `style.php` via `require_once`. Pas de fichier `.css` séparé. Chaque page ajoute son CSS spécifique dans un second `<style>`. |

   217→## Moteur Workflow
   218→
   219→Le cœur de l'application repose sur deux fonctions dans `helpers.php` :
   220→
   221→### `advance_workflow(int $submission_id)`
   222→
   223→Appelée après chaque validation pour déclencher l'étape suivante.
   224→
   225→**Algorithme** :
   226→1. Récupère toutes les étapes actives du formulaire, triées par `ordre ASC`
   227→2. Groupe par `ordre` : même `ordre` = étapes parallèles (toutes doivent être validées)
   228→3. Cherche le premier groupe sans tokens générés → génère les tokens et envoie les mails
   229→4. Si le groupe précédent n'est pas entièrement `done` → attend (ne rien faire)
   230→5. Si tous les groupes sont `done` → met `closed_at` sur la soumission + notifie l'agent par email
   231→
   232→**Exemple de workflow à 4 étapes** :
   233→```
   234→Étape 1 (ordre=1) : Responsable direct     → séquentiel
   235→Étape 2 (ordre=2) : Service informatique    → séquentiel
   236→Étape 3 (ordre=3) : RH + Logistique (ordre=3) → PARALLÈLE (2 validateurs simultanés)
   237→Étape 4 (ordre=4) : Direction               → séquentiel
   238→```
   239→
   240→### `validate_token(string $token)` → `['status' => 'ok|invalid|already_done|closed', 'data' => [...]]`
   241→
   242→Appelée quand un validateur clique sur le lien email.
   243→
   244→**Processus** :
   245→1. Vérifie le token en base (existe, pas expiré, pas déjà traité)
   246→2. Si la soumission est déjà fermée → retourne `'closed'`
   247→3. Met à jour `done_at` sur le token
   248→4. En cas de refus : ferme la soumission (`status='refuse'`), notifie l'agent
   249→5. Appelle `advance_workflow()` pour déclencher l'étape suivante si nécessaire
   250→6. Retourne le statut et les données de la soumission

   254→## Système d'authentification
   255→
   256→### `get_auth_user()` dans `helpers.php`
   257→
   258→```
   259→$_SERVER['AUTH_USER'] = "DREETS\dupont"
   260→                           ↓
   261→              get_auth_user() = "dupont@dreets.gouv.fr"
   262→```
   263→
   264→**Logique** :
   265→1. Si `AUTH_USER` est vide → affiche une page 401 stylisée (plus d'exception fatale)
   266→2. Si contient `\` (format Windows `DOMAINE\login`) → extrait le login et ajoute `@dreets.gouv.fr`
   267→3. Sinon → suppose que c'est déjà un email, passe en minuscules
   268→
   269→### Hiérarchie des rôles
   270→
   271→| Fonction | Vérification | Utilisation |
   272→|---|---|---|
   273→| `is_super_admin()` | `email === ADMIN_EMAIL` (config.php) | Accès total, suppression admin |
   274→| `is_admin_user()` | `email IN admins` (table SQLite) | Dashboard, form builder, paramètres, monitoring |
   275→| `is_form_owner($form_id)` | `email IN form_owners` pour ce formulaire | Tableau de suivi propriétaire (form_tracking.php) |
   276→
   277→### Flux de demande admin
   278→
   279→```
   280→Agent demande l'accès admin (admin_access.php)
   281→  → admin_request créée dans la table
   282→  → Email envoyé au super admin avec lien d'approbation
   283→  → Lien = page de confirmation POST (pas d'effet de bord au GET)
   284→  → Si approuvé → email ajouté à la table `admins`
   285→```

   289→## Système d'alertes
   290→
   291→### Architecture à 3 composants
   292→
   293→1. **`admin_alerts.php`** — Interface de configuration
   294→   - Champ date limite par formulaire (`deadline_field` sur `forms`)
   295→   - Règles : J-N jours avant la deadline, condition (étapes incomplètes), destinataires
   296→   - 6 cibles : admin, agent, validateurs en cours, admin+agent, admin+validateurs, email personnalisé
   297→   - Activation/désactivation, historique des alertes
   298→
   299→2. **`alert_check.php`** — Script CLI (planifier toutes les 6h)
   300→   - Parcourt les soumissions `en_cours`
   301→   - Calcule la distance à la date cible via `deadline_field`
   302→   - Évalue les conditions (étapes incomplètes)
   303→   - Envoie les emails d'alerte avec tableau récapitulatif
   304→   - Trace dans `alert_log` — déduplication : 1 alerte par règle + soumission + jour
   305→
   306→3. **Intégration monitoring** — `monitoring.php`
   307→   - Section « Alertes actives » avec code couleur urgence
   308→   - Compteur dans les statistiques globales
   309→   - Historique des dernières alertes
   310→   - Alerte si `alert_check.php` n'a pas tourné depuis > 24h

   314→## Mode Test (X-Test-Mode)
   315→
   316→Le mode test permet de faire des tests automatisés via curl ou un navigateur headless sans modifier l'application.
   317→
   318→### Activation
   319→
   320→Header HTTP : `X-Test-Mode: 1`
   321→
   322→### Comportement modifié en mode test
   323→
   324→| Comportement | Normal | Mode test |
   325→|---|---|---|
   326→| Authentification | `$_SERVER['AUTH_USER']` (IIS) | `$_SERVER['HTTP_X_TEST_USER']` |
   327→| CSRF | Vérification obligatoire | Bypass automatique |
   328→| Emails | Envoi réel via SMTP | Interceptés dans `$GLOBALS['_test_mails']` |
   329→| Base de données | `db/workflow.db` | `db/workflow_test.db` (séparée) |
   330→| Erreurs | Display + warning PHP | `E_ERROR | E_PARSE` uniquement |
   331→| Réponses | Redirections HTML | JSON pour les actions POST (erreurs, confirmations) |
   332→
   333→### Fonction `test_json_response(array $data)`
   334→
   335→Appelée dans les pages pour retourner du JSON au lieu de `die()` ou `header('Location: ...)` en mode test. Permet aux tests automatisés de vérifier le résultat sans parser du HTML.
   336→
   337→### API de test : `test_api.php`
   338→
   339→Accessible uniquement avec `X-Test-Mode: 1`. Actions disponibles :
   340→
   341→| Action | Description |
   342→|---|---|
   343→| `mails` | Liste les emails interceptés |
   344→| `reset_mails` | Vide la file d'attente |
   345→| `tokens` | Liste les tokens de validation |
   346→| `submission` | Détail d'une soumission |
   347→| `submissions` | Liste toutes les soumissions |
   348→| `cleanup` | Supprime les données de test |
   349→| `full_cleanup` | Réinitialise complète |
   350→| `forms` | Liste les formulaires |
   351→| `steps` | Liste les étapes |
   352→| `add_admin` | Ajoute un admin |
   353→| `remove_admin` | Supprime un admin |
   354→| `add_recipient` | Ajoute un destinataire |
   355→| `stats` | Statistiques globales |
   356→
   357→### Suite de tests HTTP : `test_http.php`
   358→
   359→Démarre le serveur PHP intégré et exécute 12 phases de tests via curl :
   360→0. Vérification serveur & mode test
   361→1. Configuration du workflow (destinataires + admin)
   362→2. Agent remplit le formulaire onboarding
   363→3. Validation étape 1 (Responsable direct)
   364→4. Refus étape 2 (Service informatique)
   365→5. Workflow complet bout-en-bout (4 étapes validées)
   366→6. Tokens déjà traités / invalides
   367→7. Validation des champs obligatoires
   368→8. Annulation de soumission
   369→9. Régénération de token
   370→10. Contrôle d'accès admin
   371→11. Rendu des pages HTML
   372→12. Vérification finale des statistiques

   376→## Conventions de code
   377→
   378→### Nommage
   379→
   380→- Fonctions : `snake_case` (ex: `get_auth_user()`, `advance_workflow()`)
   381→- Variables : `snake_case` (ex: `$form_fields`, `$submitted_by`)
   382→- Tables SQLite : `snake_case` (ex: `form_fields`, `step_recipients`)
   383→- Clés de settings : `snake_case` (ex: `delai_relance_h`, `relance_max`)
   384→
   385→### Structure
   386→
   387→- Pas de classes, pas d'objets (sauf PDO et PHPMailer)
   388→- Pas de namespaces, pas d'autoloading
   389→- `require_once __DIR__ . '/fichier.php'` exclusivement
   390→- Chaque page suit le même modèle :
   391→
   392→```php
   393→<?php
   394→require_once __DIR__ . '/helpers.php';
   395→// ... logique métier ...
   396→?>
   397→<!DOCTYPE html>
   398→<html lang="fr">
   399→<head>
   400→    <meta charset="UTF-8">
   401→    <title>Titre — DREETS</title>
   402→    <?php require_once __DIR__ . '/style.php'; ?>
   403→    <style>
   404→        /* CSS spécifique à cette page */
   405→    </style>
   406→</head>
   407→<body>

   454→## Tâches courantes
   455→
   456→### Ajouter un nouveau formulaire
   457→
   458→1. En back office (`admin_forms.php`), créer le formulaire (slug, libellé, description)
   459→2. Ajouter les champs — le nom technique est auto-généré à partir du libellé via `generate_field_name()`
   460→3. Configurer les étapes de validation et les destinataires
   461→4. Le formulaire est automatiquement disponible sur la page d'accueil des agents
   462→5. **Aucune modification de code nécessaire** — tout est dynamique via `form_fields`
   463→
   464→### Ajouter une section à un formulaire existant
   465→
   466→1. En back office, ajouter les champs dans le formulaire
   467→2. Utiliser un `card_group` existant ou nouveau pour le regroupement visuel
   468→3. Les données sont stockées en JSON — aucune migration SQLite nécessaire
   469→4. Les mails et le dashboard affichent automatiquement les nouveaux champs
   470→
   471→### Ajouter un type de champ
   472→
   473→1. Ajouter le type dans le sélecteur de `admin_forms.php`
   474→2. Ajouter le rendu dans la fonction `render_field()` de `form.php`
   475→3. Le stockage en JSON est agnostique du type — pas de migration nécessaire
   476→
   477→### Modifier le workflow
   478→
   479→1. Le workflow est entièrement piloté par les tables `steps` et `step_recipients`
   480→2. Même `ordre` = étapes parallèles, `ordre` différent = séquentiel
   481→3. Les modifications n'affectent que les nouvelles soumissions

   544→## Points d'attention
   545→
   546→| Point | Détail |
   547→|---|---|
   548→| **SQLite et concurrence** | `PRAGMA journal_mode=WAL` activé — lectures simultanées OK. Écritures séquentielles. Suffisant pour < 100 soumissions/jour. |
   549→| **Tokens** | `random_bytes(32)` — cryptographiquement sûrs, non rejouables, avec expiration configurable. |
   550→| **Auth AD** | `AUTH_USER` peut être vide si IIS n'est pas configuré en Windows Auth → page 401 stylisée au lieu d'une erreur fatale. |
   551→| **SMTP sans TLS** | Configuré pour un SMTP intranet sans authentification. Si TLS/auth requis, configurer via `admin_settings.php`. |
   552→| **Config protégée** | `config.php` n'est jamais écrasé par `update.ps1` — le déploiement préserve la configuration locale. |
   553→| **fputcsv()** | PHP 8.4+ exige un 5e paramètre. `export_csv()` utilise `fputcsv($f, $row, ';', '"', '\\')`. |
   554→| **PHP built-in server** | Instable avec cette application (crash après 1-2 requêtes). Utiliser le mode test + subprocess pour les tests automatisés. |
   555→| **Zéro JS** | L'application est 100% sans JavaScript depuis la v3.0.0. Toutes les interactions sont côté serveur (confirmations via confirm_action.php, onglets via GET, toggles via liens). |
